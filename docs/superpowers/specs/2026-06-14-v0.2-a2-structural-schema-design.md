# v0.2 / A2 — structural schema + member enrichment (design)

**Date:** 2026-06-14
**Status:** approved approach, spec under review
**Scope:** Sub-project A2 of v0.2. Lay down migration `0002_structure.sql` for the
structural domains (Riigikogu terms, sessions/sittings, committees, electoral districts)
and enrich the `members` and `votes` records from the official API. **Data layer only** —
the member-detail web UI (vote timeline, party-switch lines) belongs to sub-project B with
the design-system foundation and is explicitly out of scope here.

Builds directly on A1 (the API cutover). A1's raw cache (`apps/scraper/cache/api/`) already
contains everything A2 needs except the sessions list and the photo binaries.

---

## Why

A1 rebased ingestion onto the API but kept the v0.1 schema (parties, members,
member_party_terms, votes, ballots). The API carries far more per-member and per-vote
structure than v0.1 models. A2 lands the structural tables now — "schema laid down up
front so later UI is additive, not migratory" (governing decision 2) — and enriches the
member record so sub-project B can build member pages against real data.

Crucially, almost all of A2 is reproducible **offline** from the cache A1 already
committed: `plenary-members.json` carries `dateOfBirth`, `dateOfDeath`, `gender`,
`parliamentSeniority`, `email`, `phone`, `photo`, per-term `electoralDistrictHistory`, and
`committees[]` (with membership start/end + role); `votings.jsonl` carries each voting's
`sitting{}` and `relatedDraft{}`. The cutover's pydantic models dropped the latter two via
`extra='ignore'`, but the raw JSONL retains them — so votes enrichment is a pure re-read,
no re-fetch.

Two pieces are **not** in the cache and require a one-time live fetch (then archived, so
`rebuild` stays offline): the sessions list (`/api/sessions`) and the member photo
binaries (`/api/files/{uuid}/download`).

## Verified API shapes (probed live, 2026-06-14)

- `GET /api/plenary-members` → array (101) of members, each with top-level keys:
  `uuid, fullName, firstName, lastName, active, email, phone, upn, gender (MALE|FEMALE),
  dateOfBirth, dateOfDeath, parliamentSeniority (int, e.g. 3354), agesInParliament[],
  electoralDistrictHistory[], photo{uuid,fileName,fileExtension,_links.download.href},
  plenaryMembership{uuid,membershipNumber,startDate,endDate,role,jobTitle},
  factions[] (type.code = FRAKTSIOON only), committees[]`.
  - `committees[]` element: `{uuid, name, type{code: ALALINE_KOMISJON|ERIKOMISJON, value},
    active, membership{membershipNumber, startDate, endDate, role{code,value}}}`. 99/101
    members have at least one committee.
  - `electoralDistrictHistory[]` element: `{membership (int term no.),
    electoralDistrict{code, value}}`. 13 distinct districts in term 15.
- `GET /api/sessions` → array (176 across all terms; 16 for term 15) of
  `{membership (int term no.), number (int), type{code: KORRALINE|ERAKORRALINE, value},
  startDate, endDate}`. KORRALINE sessions are numbered 1,2,3…; ERAKORRALINE 101+.
  KORRALINE ranges are disjoint; ERAKORRALINE ranges (often single-day) overlap the
  enclosing KORRALINE session.
- `GET /api/volumes/{sitting-uuid}` (plenary sitting volume) → `{uuid, title, volumeType:
  plenarySittingVolume, membership, year, created, …}`. **No session reference** — so
  sitting→session is resolved by date containment, not a direct link.
- Each voting in `votings.jsonl` carries `sitting{uuid, title, volumeType, created}` and
  (395/598) `relatedDraft{uuid, title, mark, copyOrOriginal}`.
- `GET /api/files/{photo-uuid}/download` → image bytes (~100 KB JPEGs).

## Decisions (this session)

1. **Sessions:** re-fetch `/api/sessions` once and archive to `cache/api/sessions.json`;
   model `sessions` + `sittings`. (Chosen over deferring or date-heuristic derivation.)
2. **Migrations:** keep numbered raw `.sql`, add a `schema_migrations` tracking table and a
   small auto-apply helper.
3. **Photos:** download + compress to thumbnails committed to the repo (used in the members
   list); full-res served at runtime from the API URL on the profile page.
4. **Bill link:** capture `relatedDraft` on `votes` now (`draft_uuid` + title/mark) as the
   v0.3 vote→bill→Eurovoc bridge; no bills table yet.
5. **Boundary:** A2 is data-layer only; member-detail UI is sub-project B.
6. **History depth:** ingest the **full** `electoralDistrictHistory` and committee history
   present in the member payload (cheap, already cached), not current-term-only.

## Schema — migration `0002_structure.sql`

Raw SQL, idempotent (`CREATE TABLE IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`), same style
as 0001. Wrapped in a single `BEGIN/COMMIT`.

### Versioning

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  version    TEXT PRIMARY KEY,            -- '0001', '0002'
  applied_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
INSERT INTO schema_migrations (version) VALUES ('0001'), ('0002')
  ON CONFLICT DO NOTHING;
```

### New tables

```
riigikogu_terms (
  id          SERIAL PK,
  number      INT  NOT NULL UNIQUE,       -- koosseis no., e.g. 15
  started_on  DATE,                       -- seeded for 15; NULL for back-referenced terms
  ended_on    DATE
)

sessions (                               -- istungjärk
  id          SERIAL PK,
  term_id     INT  NOT NULL REFERENCES riigikogu_terms(id),
  number      INT  NOT NULL,
  type_code   TEXT NOT NULL,              -- KORRALINE | ERAKORRALINE
  started_on  DATE NOT NULL,
  ended_on    DATE,
  UNIQUE (term_id, number)
)

sittings (                               -- istung
  id             SERIAL PK,
  riigikogu_uuid UUID NOT NULL UNIQUE,
  session_id     INT  REFERENCES sessions(id),   -- NULL if date maps to no session
  term_id        INT  REFERENCES riigikogu_terms(id),
  title          TEXT,
  sitting_date   DATE NOT NULL
)

committees (
  id             SERIAL PK,
  riigikogu_uuid UUID NOT NULL UNIQUE,
  name           TEXT NOT NULL,
  type_code      TEXT NOT NULL            -- ALALINE_KOMISJON | ERIKOMISJON
)

member_committee_terms (
  id           SERIAL PK,
  member_id    INT  NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  committee_id INT  NOT NULL REFERENCES committees(id),
  role_code    TEXT,                      -- LIIGE, ESIMEES, ASEESIMEES, …
  started_on   DATE,
  ended_on     DATE
)

electoral_districts (
  id    SERIAL PK,
  code  TEXT NOT NULL UNIQUE,             -- JARVA_JA_VILJANDIMAA
  name  TEXT NOT NULL                     -- Järva- ja Viljandimaa
)

member_district_terms (
  id          SERIAL PK,
  member_id   INT NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  district_id INT NOT NULL REFERENCES electoral_districts(id),
  term_id     INT NOT NULL REFERENCES riigikogu_terms(id),
  UNIQUE (member_id, district_id, term_id)
)
```

Indexes: `sittings(session_id)`, `sittings(sitting_date)`, `member_committee_terms(member_id)`,
`member_district_terms(member_id)`, `sessions(term_id)`.

### Member enrichment (ALTER members)

`date_of_birth DATE`, `date_of_death DATE`, `gender TEXT`, `email TEXT`, `phone TEXT`,
`parliament_seniority_days INT` (the API's `parliamentSeniority`; labelled days — the
conventional reading of the integer), `mandate_started_on DATE`
(`plenaryMembership.startDate`), `photo_uuid UUID`, `photo_file_name TEXT`,
`photo_url TEXT` (full download href), `photo_thumb_path TEXT` (repo-relative thumbnail).

### Vote enrichment (ALTER votes)

`sitting_id INT REFERENCES sittings(id)`, `draft_uuid UUID`, `draft_title TEXT`,
`draft_mark TEXT` (stored TEXT for safety though the sample is integer). Index
`votes(sitting_id)`, `votes(draft_uuid)`. Discipline views from 0001 are **untouched** —
scores must not move (A1 parity numbers stand: 21024 counted / 20940 aligned / 84
defections).

## sitting → session mapping (writer logic)

For each sitting, `sitting_date = min(voting.startDateTime::date)` across its votings.
Choose the session whose `[started_on, ended_on]` contains `sitting_date`; on overlap
prefer the **narrower** range (single-day ERAKORRALINE wins over the enclosing KORRALINE);
no match → `session_id = NULL`. Deterministic and offline.

## Photo pipeline (new CLI `photos`, network-gated)

For each member with a `photo.uuid`: GET `/api/files/{uuid}/download` (through `ApiClient`,
1 req/s, configured UA), compress to a small thumbnail (~160 px longest edge, WebP) with
**Pillow** (new scraper dependency), write to
`apps/web/public/members/<member_uuid>.webp` (committed to git, served statically by
Next.js), and set `members.photo_thumb_path`. `daily`/`rebuild` set the photo
*metadata + url* from cached member JSON offline; thumbnails refresh only when `photos`
runs. Full-res images are never stored — the profile page loads `photo_url` at runtime.

## Ingestion code changes

- **`api_models.py`** — extend `PlenaryMember` (`gender`, `dateOfBirth`, `dateOfDeath`,
  `parliamentSeniority`, `committees[]`, `electoralDistrictHistory[]`,
  `plenaryMembership.startDate`); add `sitting` + `relatedDraft` to `Voting`; new models
  `Session`, `Committee`/`CommitteeMembership`, `DistrictHistory`, `RelatedDraft`, `Sitting`.
- **`api_cache.py`** — archive/read `sessions.json`.
- **`api_client.py`** — `get_sessions()` and `download_file(uuid) -> bytes`.
- **`api_parse.py`** — helpers as needed (e.g. role/type code passthrough); no change to
  `vote_type_slug` / `decision_to_choice` (scores frozen).
- **`writer.py`** — populate all new tables; term upsert-by-number; sitting→session date
  map; committee/district term building; member + vote enrichment upserts.
- **`db.py`** — `apply_migrations()` that runs unapplied `NNNN_*.sql` in order against
  `schema_migrations`; new upsert helpers for the new tables.
- **`cli.py`** — `backfill` and `members` also fetch+archive `sessions.json`; `rebuild`
  populates every new table from the archived cache with no network; new `photos`
  subcommand.

## Testing (offline)

Extend `apps/scraper/fixtures/api/` and add `pytest` cases:

- sessions date-mapping, including the ERAKORRALINE/KORRALINE overlap tiebreak and the
  no-match → NULL case;
- committee-term and district-term construction from a member payload;
- member enrichment field mapping (birth/death/gender/seniority/mandate/photo metadata);
- vote → sitting and vote → draft linkage from `votings.jsonl`;
- `schema_migrations` idempotency (re-running 0002 is a no-op; versions recorded);
- photo compression on a tiny fixture image with the network mocked (asserts a written,
  size-bounded WebP and a recorded `photo_thumb_path`).

`ruff` clean across the package. Existing A1 tests continue to pass unchanged.

## Validation plan

1. Offline: `rebuild` from the existing cache + a newly-archived `sessions.json` populates
   all new tables; spot-check counts (≈150 sittings, 16 term-15 sessions, committee and
   district term rows for ~99 members) and that discipline view outputs are byte-identical
   to A1.
2. Network (local/Actions, not sandbox): run the sessions fetch and `photos` once; commit
   `sessions.json` + thumbnails; re-run offline `rebuild` to confirm reproducibility.
3. Production: apply `0002` via `apply_migrations()`, then `rebuild`. Discipline numbers
   must be unchanged from A1.

## Out of scope (A2)

Member-detail web UI (→ B); bills/speeches/Eurovoc tables (v0.3/0.6); committee *votings*
(`/api/votings/committees`) and the `/api/statistics/*` precomputed endpoints; party /
committee cohesion rollups (v0.4); any change to the discipline metric or its views.
