# v0.2 / B ‚Äî design-system foundation + members table (design)

**Date:** 2026-06-15
**Status:** approved approach, spec under review
**Scope:** Sub-project B of v0.2. Stand up the design-system foundation in `apps/web`
(tokens, party palette, shadcn/ui, layout shell + nav, motion + View Transitions) and
rebuild the v0.1 members list as a polished, sortable/filterable enriched table against
it. **The per-member detail page (vote timeline, party-switch lines, visx) is the next
slice (C) and is explicitly out of scope here** ‚Äî but the foundation is built so that page
drops in additively.

Builds on A1 (API cutover) and A2 (structural schema + member enrichment, live in prod).
A2 already populated the data B surfaces: member photos (`public/members/<uuid>.webp`,
`photo_thumb_path`), party, bio, committees, districts. No scraper or schema changes in B ‚Äî
this slice is `apps/web` only, plus one small read-side query extension.

---

## Why

The web app today is a single server-rendered page with a bare Tailwind table: no
component system, no design tokens, no party palette, no motion. Governing decision 4 of
the roadmap is "UI is first-class from v0.2." B establishes the design layer every
subsequent version (v0.3 topics, v0.4 rollups, v0.5 speeches, ‚Ä¶) ships against, so later
UI is additive rather than a re-skin. The members table is the first real application of
the system and the immediate user-visible upgrade.

## Locked decisions (from brainstorming, 2026-06-15)

1. **Visual direction: Editorial / broadsheet.** Serif display + headings, warm paper
   background, hairline rules, generous whitespace, party color used as a restrained
   accent (outlined pill + thin discipline bar), not large fills.
2. **Dark mode: both from the start.** All design values are CSS variables with light and
   dark themes; honor `prefers-color-scheme` with a manual toggle.
3. **List view: enriched table** ‚Äî sortable/filterable, photo + name, party badge,
   discipline bar + tabular %, counted votes, defections; filter by party. (Not a card
   grid, not a toggle.)
4. **Party palette (locked hex):**

   | Party | Fill | Ink (light: pill text/border) | Dark (lifted fill) |
   | --- | --- | --- | --- |
   | RE (Reform) | `#F5C518` | `#8A6D00` | `#F5C518` |
   | EKRE | `#16315E` | `#16315E` | `#5B86C4` |
   | KE (Keskerakond) | `#00A35B` | `#007A43` | `#2BC077` |
   | E200 (Eesti 200) | `#16B6BE` | `#0C7E84` | `#2CC9D1` |
   | SDE | `#E4002B` | `#C20021` | `#FF3355` |
   | I (Isamaa) | `#1F5AA8` | `#16508F` | `#5B97DA` |

   Party color is always paired with the name/abbreviation, so it never carries meaning
   alone (keeps SDE-red vs KE-green and the three blues readable for color-blind users).

## Stack additions (on top of the existing Next.js 15 App Router + React 19 + Tailwind v4 + next-intl base)

- **shadcn/ui** (Radix primitives + Tailwind v4). Copy-in components, themed via our CSS
  variables. Initialize with the Tailwind-v4 / React-19 compatible setup. Only the
  components B actually uses are added (see Components).
- **next-themes** ‚Äî `class`-strategy dark mode with a manual toggle and system default.
  `suppressHydrationWarning` on `<html>`; `ThemeProvider` wraps the app.
- **next/font** ‚Äî load the two families self-hosted (no external requests, good CLS):
  a serif for display/headings (**Source Serif 4**) and a sans for UI/body (**Inter**),
  exposed as `--font-serif` / `--font-sans`. Tabular figures for all numeric cells.
- **Framer Motion** ‚Äî component-level motion only in B: members-table row reorder on
  sort, subtle list/section entrance, hover affordances. Gated behind
  `prefers-reduced-motion`.
- **View Transitions** ‚Äî enable Next.js View Transitions so the eventual list -> detail
  navigation (slice C) is coherent. In B we wire the config and use it for the in-page
  transitions that exist (theme/sort), so C inherits it for free.
- **Recharts / visx are NOT added in B.** They arrive with the charts that need them
  (timeline in C, rollups in v0.4+). Adding them now would be unused surface area.

## Token system

All tokens live as CSS variables in `app/[locale]/globals.css`, surfaced to Tailwind v4
via `@theme inline` so they're usable as utility classes and by shadcn. Two layers:

- **Primitive / semantic UI tokens** ‚Äî background (warm paper `#FAF8F3` light / near-black
  `#16140F` dark), foreground/ink, muted, border (hairline), card, accent, ring; a spacing
  and radius scale; an elevation scale (shadcn surfaces stay nearly flat to match the
  editorial direction ‚Äî borders over shadows). Light values under `:root`, dark under
  `.dark`.
- **Party tokens** ‚Äî for each party a fill, ink, and the dark-lifted fill, e.g.
  `--party-re-fill`, `--party-re-ink`, plus the `.dark` overrides. The mapping from the DB
  `party_short_name` (RE / EKRE / KE / E200 / SDE / I) to these variables lives in one
  place, `lib/party.ts` (`partyToken(shortName)` -> a small object / set of class names),
  so no component hard-codes a hex. Unknown/non-attached (`null` party) falls back to a
  neutral token.

The party palette and the token contract are the durable artifact of B: later versions
add components, not new color decisions.

## Components and boundaries

New `apps/web/components/` directory. Each unit has one purpose, plain typed props, and is
understandable without reading its consumers.

- `components/ui/*` ‚Äî shadcn primitives actually used: `button`, `badge`, `table`,
  `dropdown-menu` (party filter), `toggle` / `tooltip` as needed. Themed by the tokens
  above; not modified beyond shadcn defaults except to consume our variables.
- `components/site-header.tsx` ‚Äî the editorial masthead/nav shell: serif wordmark, nav
  links (Liikmed active; Fraktsioonid/Teemad as inert placeholders for later versions),
  `LocaleToggle`, `ThemeToggle`. Server component; toggles are small client islands.
- `components/theme-toggle.tsx` (client) ‚Äî next-themes light/dark/system control.
- `components/locale-toggle.tsx` (client) ‚Äî ET/EN switch using next-intl `Link`/router,
  preserving the current path.
- `components/party-badge.tsx` ‚Äî outlined editorial pill; takes `shortName`, resolves via
  `lib/party.ts`. Pure, no data access.
- `components/discipline-bar.tsx` ‚Äî the track + fill + tabular %; takes `score` (0..1 or
  null) and `shortName` for the fill color. Pure.
- `components/member-avatar.tsx` ‚Äî `next/image` from `photo_thumb_path`
  (`/members/<uuid>.webp`), with an initials fallback colored by party token when a photo
  is missing. Pure.
- `components/members-table.tsx` (client) ‚Äî owns sort state (by discipline / name /
  counted / defections) and the party filter; renders rows from props using the pure
  components above; Framer Motion animates reorder. Receives already-fetched rows as
  props ‚Äî **no data access in the client**.
- `app/[locale]/page.tsx` ‚Äî server component: fetches rows, renders `SiteHeader` + the
  table section heading/lede + `<MembersTable rows={‚Ä¶} />` + footer. Keeps the existing
  `revalidate = 3600` ISR and the DB-error fallback.
- `app/[locale]/layout.tsx` ‚Äî adds `ThemeProvider`, the font variables on `<html>`,
  `suppressHydrationWarning`, and the warm-paper/ink body tokens.
- `lib/party.ts` ‚Äî the single party-token mapping + small helpers.

## Data flow

Unchanged architecture: server reads, client only renders. `getMemberDiscipline()` gains
two columns so the table can render avatars and (future) detail links without a second
query:

- add `photo_thumb_path` (already on `members`) to the SELECT and the
  `MemberDisciplineRow` type;
- `slug` is already selected ‚Äî kept for the eventual `/[locale]/members/[slug]` link in C
  (rendered as a non-navigating row in B, or a link to a 404-safe route guarded until C;
  default: name is plain text in B, wrapped in a link in C).

No new query, no schema change. Rows flow server -> `page.tsx` -> `MembersTable` as typed
props. Sorting/filtering is pure client state over that in-memory array (84‚Äì102 rows; no
pagination needed).

## i18n

All new chrome strings (nav labels, theme/locale toggle a11y labels, sort/filter labels,
"all parties") go through next-intl in `messages/et.json` + `messages/en.json`; ET is
default. Party abbreviations (RE/EKRE/‚Ä¶) are proper nouns ‚Äî not translated. The existing
`members.*` / `site.*` / `footer.*` keys are reused; new keys added under a `nav.*` and
`table.*`/`filter.*` namespace.

## Motion & accessibility

- Motion is purposeful and fast (‚â§ ~200ms); every animation is disabled (or reduced to an
  opacity fade) under `prefers-reduced-motion: reduce`.
- Radix gives keyboard nav + focus management for the dropdown/toggles for free; ensure a
  visible focus ring (token `--ring`) on all interactive elements.
- Contrast: ink tones chosen for AA on warm paper; the lifted dark tones for AA on
  near-black. Discipline bars and party color are always accompanied by text.
- Photos get meaningful `alt` (member name); decorative chrome is `aria-hidden`.

## Testing / verification

- **Type + lint:** `pnpm --filter web typecheck` and `lint` clean.
- **Unit (vitest):** `lib/party.ts` mapping (every short name -> expected tokens; null ->
  neutral); the sort comparator (discipline asc with nulls last, name tiebreak ‚Äî matches
  the current SQL `ORDER BY`); the filter (by party, "all"). Pure functions, no DOM.
- **Manual / visual (run the app):** members table renders at mobile + desktop widths;
  sort and party filter work; light/dark toggle and ET/EN toggle work and persist;
  `prefers-reduced-motion` honored; keyboard nav through header + filter; DB-error
  fallback still shows. Lighthouse a11y pass (contrast on all six party tokens, focus
  order).
- **No data regression:** the table's default order and numbers match today's page (same
  `getMemberDiscipline` ordering); B changes presentation, not the metric.

## Risks / notes

- **Tailwind v4 + shadcn:** shadcn's v4 path uses `@theme inline` and CSS-variable tokens
  rather than `tailwind.config` theme keys; the init must target v4 so generated components
  consume our variables. Verify the first added component renders themed before building
  the rest.
- **next-themes hydration:** requires `suppressHydrationWarning` on `<html>` and the
  provider high in the tree to avoid a flash; validate no hydration warning in dev.
- **Three blues (EKRE/E200/Isamaa):** locked hex were chosen for separation; re-check on a
  real screen during implementation, especially dark mode.
- **Editorial fonts add weight:** two self-hosted families; subset to Latin + Estonian
  diacritics (√µ √§ √∂ √º ≈° ≈æ) and load only needed weights to keep payload small.
