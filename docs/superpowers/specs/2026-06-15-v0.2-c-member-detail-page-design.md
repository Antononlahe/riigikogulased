# v0.2 / C — member-detail page (design)

**Date:** 2026-06-15
**Status:** approved approach, spec under review
**Scope:** Sub-project C of v0.2. A per-member detail page at `/[locale]/members/[slug]`
built on B's design system: member header with the "party member, not in faction" indicator,
discipline summary, a **per-party-at-time discipline breakdown**, the signature **vote
timeline** (combined monthly trend + per-vote strip + party-switch markers, built with
**visx**), and an affiliations/bio panel from the A2-enriched member record. The members-list
rows become links into it. No scraper or schema changes — this slice is `apps/web` only, plus
read-side query additions.

Builds on A1 (API cutover), A2 (structural schema + enrichment), B (design system), and the
erakond reconciliation slice (which added `member_current_party.in_faction` and the
faction-first/erakond-fallback scoring this page surfaces).

---

## Why

B delivered the list and the design foundation but deliberately deferred the per-member page.
C is the first place the dashboard tells an individual story: how a member votes relative to
their party line over time, when they switched benches, and who they are. It is also where the
erakond slice's **"party member but not in a faction"** distinction becomes visible (the
roadmap's "noticeable on the individual MP page" requirement). All data already exists in the
DB; C is additive UI + read queries.

## Locked decisions (from brainstorming, 2026-06-15)

1. **Scope: full.** Header + discipline summary + per-party-at-time breakdown + combined vote
   timeline + affiliations/bio panel.
2. **Timeline representation: combined.** A monthly discipline-% trend on top and a dense
   per-vote strip below, sharing one time axis, with party-switch vertical guides and a
   per-tick hover tooltip.
3. **Charting library: visx** (`@visx/scale`, `@visx/shape`, `@visx/axis`, `@visx/tooltip`,
   `@visx/group`) — the roadmap's named lib; React-first primitives reused later by the v0.4
   agreement matrix and hall-plan. The timeline is the only client/visx island; everything
   else is server-rendered with plain typed props.

## Architecture / rendering

- **Route:** `app/[locale]/members/[slug]/page.tsx`, a server component.
  `generateStaticParams` enumerates every `members.slug`; `export const revalidate = 3600`
  (ISR, matching the list). An unknown slug calls `notFound()`.
- **List → detail:** the member names in `components/members-table.tsx` (plain text in B)
  become locale-aware `next-intl` `Link`s to `/members/[slug]`, with a Next.js **View
  Transition** for the morph. Honor `prefers-reduced-motion`.
- **Data flow:** one server query `getMemberDetail(slug)` returns fully-typed props; the page
  shapes nothing heavy itself. The timeline component receives plain arrays (no DB access in
  the client).

## Faction-status chip (the erakond "noticeable" indicator)

Driven by `member_current_party.in_faction` + the scoring party:

- In a fraktsioon (normal): no chip (the party badge already says it).
- Scoring party from erakond, not in a fraktsioon: a visible chip, e.g. ET
  "Reformierakonna liige · ei kuulu fraktsiooni" / EN "Reformierakond member · not in a
  faction group".
- No scoring party at all (truly independent / non-parliamentary party): a muted
  "fraktsioonitu" / "unaffiliated" state, and the discipline summary explains the member is
  not scored.

## Data / queries

`getMemberDetail(slug)` (server, in `lib/queries.ts`) returns:

1. **Member record** — `members` row (name, slug, photo_thumb_path, photo_url, date_of_birth,
   gender, email, phone, parliament_seniority_days, mandate_started_on) joined to
   `member_current_party` (party_short_name, party_name, **in_faction**).
2. **Discipline summary** — from `member_discipline` (counted_votes, aligned_votes,
   defections, derived score).
3. **Per-party-at-time breakdown** — `member_vote_alignment` joined to `votes`, filtered to
   non-procedural scored ballots, **grouped by `party_id`** (the scoring party at each vote),
   each group → `{party_short_name, party_name, counted, aligned, defections, first_date,
   last_date}`. One row for non-switchers; several for members who changed scoring party
   mid-window (e.g. Põlluaas: EKRE→ISAMAA).
4. **Per-vote series** for the timeline (~600 rows) — `member_vote_alignment` joined to
   `votes`, returning the raw fields `{voted_at, title, draft_title, member_choice,
   party_majority_choice, is_procedural, party_short_name}`. The query does NOT classify;
   `classifyVote()` in `lib/member-detail.ts` (pure, unit-tested) derives
   `classification ∈ {aligned, against, excluded}`: `excluded` if `is_procedural` OR
   `member_choice` not in (yes,no,abstain) OR `party_majority_choice` IS NULL; else `aligned`
   if `member_choice = party_majority_choice`; else `against`. Keeping the rule in one tested
   function (not SQL) avoids drift. The series is trimmed to the chart's needed fields to keep
   the SSR payload lean (~600 small objects).
5. **Party-switch markers** — dates within the window where the member's scoring party
   changes, derived from `member_faction_terms` + `member_erakond_terms` term boundaries (or
   equivalently from `party_id` changes between consecutive votes in the series). Each marker:
   `{date, from_party_short, to_party_short}`.
6. **Affiliations** — `member_committee_terms`→`committees` and
   `member_district_terms`→`electoral_districts` (existing tables), current + past.

The **monthly trend** is aggregated client-side from the per-vote series (no extra query):
per calendar month, `discipline% = aligned / (aligned + against)` over non-excluded votes.

## The vote timeline (visx)

One client component, `vote-timeline.tsx`, receiving the per-vote series + switch markers +
the party palette tokens as props. Layout (top to bottom, shared `scaleTime` x-axis):

- **Trend band:** an area/line (`@visx/shape` `AreaClosed`/`LinePath`) of monthly discipline
  %, `scaleLinear` y in [0,1]. Subtle fill in the scoring-party color.
- **Per-vote strip:** one mark per vote positioned by `voted_at`; color = with-party (party
  fill token), against (a single contrast/"defection" token, shared across parties for
  legibility), excluded (muted). Marks are thin ticks; on dense months they overlap into a
  band (honest density).
- **Party-switch guides:** vertical lines at each marker date with a small label.
- **Interaction:** `@visx/tooltip` — hovering a vote shows date, title (truncated), the
  member's choice, and the party line + outcome (with/against/excluded). Keyboard: the strip
  is focusable and arrow-navigable; each mark has an `aria-label`.
- **Responsive:** width from a `ResizeObserver`/parent measure; fixed heights per band; on
  mobile the strip stays full-width and the affiliations panel stacks below.
- **Reduced motion:** entrance animation and the View Transition are skipped when
  `prefers-reduced-motion`.

## Accessibility

The timeline is decorative-plus-data; the same information is available textually via the
**per-party breakdown** table and an `aria` summary ("335 votes, 100% with party, 0 against,
Jun 2025–Jun 2026"). Marks are keyboard-focusable with labels. Color is never the only signal
(tooltip + labels carry meaning). AA contrast via the B token palette (dark tones already
lifted).

## i18n

New `memberDetail` namespace in `messages/et.json` (default) + `messages/en.json`: section
headings, the faction-status chip strings, breakdown column labels, timeline legend/tooltip
labels, affiliation labels, and an empty/not-scored state. ET is authoritative; EN mirrors.

## Testing

- **Vitest unit tests** for `lib/member-detail.ts`: classification mapping (all six choice ×
  line combinations), monthly aggregation (month bucketing, excluded votes ignored,
  empty-month handling), party-switch derivation (no switch, one switch, multiple), and
  Estonian-collation/date edge cases.
- Typecheck, `next lint`, production build (static pages generated for member slugs,
  0 `MISSING_MESSAGE`), 9+ existing tests stay green.
- Interactive checks (timeline hover/keyboard, View Transition, responsive, real data) need a
  reachable DB and are confirmed at deploy, as with B.
