# v0.2 — erakond ↔ fraktsioon reconciliation (design)

**Date:** 2026-06-15
**Status:** approved approach, spec under review
**Scope:** A data-accuracy slice of v0.2 that sits **between B (shipped) and C (member
detail page)**. Make the discipline metric reflect each member's **party (erakond)
membership**, not merely their **parliamentary faction (fraktsioon)**. Introduce a second
ingestion source — the äriregister (RIK) political-party member registry — to supply
erakond data the Riigikogu API does not carry. The visible "party member but not in a
faction" indicator on the individual MP page is **deferred to C**; this slice produces the
data and the corrected scoring that C will surface.

Builds on A1 (API cutover), A2 (structural schema), B (design system). The web change here
is intentionally minimal (correct the party badge + expose the new fields); the charting/MP
page is C.

---

## Why

The Riigikogu Open Data API only carries **fraktsioon** membership (`voters[].faction`,
member `factions[]`). It has **no party/erakond field** — verified across every key of the
member record. Our pipeline therefore equates fraktsioon with party, which is correct for
~85 of 102 members but wrong for everyone whose faction status and party membership diverge.

**Observed failure:** 17 of 102 members (~17%) are reported as
`Fraktsiooni mittekuuluvad Riigikogu liikmed` (non-attached) for the **entire** window
(2025-06-16 → 2026-06-11) in every one of their ballots. Our writer stores that as
`party_id = NULL`, and the discipline views exclude `party_id IS NULL` members entirely —
so they show **0 counted votes** and vanish from the core metric. Several are prominent
(Tanel Kiik, Jaak Aab, Tõnis Mölder, Henn Põlluaas, Leo Kunnas, Peeter Ernits).

This is **not a code bug** — the pipeline faithfully records what the API says. The API is
simply missing a fact. Example: **Alar Laneman** is recorded as non-attached to any
fraktsioon, but per the official party registry he has been an **Eesti Reformierakond**
member since **2024-07-12** (previously EKRE, 2019-03-27 → left 2024-06-14). Our window is
entirely RE-era, yet he scores 0.

The journalism goal is loyalty to a **party line**. A sitting party member who has left the
faction bench is still a party member; their alignment with the party is meaningful and
should be measured. Conversely, an MP who sits on a party's faction bench without being a
card-carrying member should still be measured against that faction's line.

## Source investigation (done during brainstorming, 2026-06-15)

- **Riigikogu member record** — only `factions[]`; no erakond field. Cannot fix this.
- **Äriregister open-data (avaandmed) API** — company data only; no party-membership
  endpoint, and **no bulk party-members download** exists on the open-data portal.
- **Äriregister party-registry web UI** (`ariregister.rik.ee/est/political_party/...`) —
  **server-side-rendered HTML**, the only available source:
  - `…/members_search?person_name=<name>` returns a parseable result list: full name, date
    of birth (`DD.MM.YYYY`), current party, and a numeric RIK person id used in the history
    link. Name search is fuzzy/prefix — the exact person is selected by **exact name +
    date of birth**.
  - `…/member_history/<person_id>` returns the **dated** membership history per party
    (party name + registration code, joined date, left date, status).
  - Verified end-to-end for Laneman: search → person id `9000034247` → RE since 2024-07-12.

## Locked decisions (from brainstorming, 2026-06-15)

1. **Build the full importer now** (not a manual override table). The importer pulls erakond
   membership for **all** members, not just the non-attached, so the data also powers C's MP
   page indicator and future toggles.
2. **Scoring-party resolution rule.** For a member M at the time of vote V:
   - If M is in a fraktsioon at V → M's scoring party = that **fraktsioon's party**
     (fraktsioon is authoritative when present — covers "in EKRE's faction but not an EKRE
     party member" → still scored against EKRE).
   - Else (non-attached) → M's scoring party = M's **erakond** at V, from äriregister.
   - Else (non-attached and no party) → **excluded** (truly independent; correct).
3. **Party line is defined by faction members only.** A party's majority position for a vote
   is the strict majority among the **fraktsioon members** of that party. Non-attached party
   members (e.g. Laneman) are scored **against** that line but do **not** shape it. Self is
   excluded from the line only when the scored member is themselves a faction member of that
   party. (Numerically near-identical to including everyone, since majorities are robust,
   but conceptually correct and matters for small parties.)
4. **Party identity keyed on registration code** (RE `80043147`, EKRE `80040344`, KE, E200,
   SDE, I, …), stable across a party's historical name changes; party name as a fallback.
5. **Rename `member_party_terms` → `member_faction_terms`** for clarity (it has always meant
   the fraktsioon-derived party). A new `member_erakond_terms` holds party membership.
6. **Web changes minimal in this slice.** The members-table badge shows the **scoring
   party** (so the 17 appear with their party and real scores); the `in_faction` /
   erakond-vs-faction data is exposed for C. No new visuals here; the "not in faction"
   indicator is C's deliverable.

## The four cases (truth table)

| In fraktsioon? | Erakond member? | Scoring party | Defines party line? |
| --- | --- | --- | --- |
| Yes, party X | Yes, X | X | Yes (as faction member of X) |
| Yes, party X | No / different | **X** (faction wins) | Yes (as faction member of X) |
| No (non-attached) | Yes, Y | **Y** (erakond) | No (scored vs Y, doesn't shape it) |
| No (non-attached) | No | excluded | n/a |

## Architecture

```
api.riigikogu.ee (JSON, 1 req/s) ─┐
                                  ├─> apps/scraper ─> Postgres (Neon) ─> apps/web (read)
ariregister.rik.ee (HTML, NEW)  ─┘
   political_party/members_search + member_history
```

The scraper gains a second source. Raw HTML from RIK is archived per-member under
`apps/scraper/cache/ariregister/` (git-committed) so the offline `rebuild` reproduces
erakond terms with no network, exactly as the API cache does for votes.

### Component 1 — äriregister client + parser (`ariregister.py`)

- `search_person(name) -> list[Candidate]` — fetch the search page, parse rows into
  `Candidate(full_name, date_of_birth, party_name, person_id | None)`. Handles pagination
  only as far as needed to find the exact match (the exact name+DOB is typically on page 1).
- `fetch_history(person_id) -> list[Membership]` — parse the history table into
  `Membership(party_name, registry_code, started_on, ended_on | None, status)`.
- **Matching:** `resolve_member(member) ->` person id, by exact `full_name` **and**
  `date_of_birth` (we store `members.date_of_birth` from A2). No match → `None`, logged,
  non-fatal. Members with a confirmed faction party do not strictly need a match (faction
  wins); only the non-attached critically need one, so an imperfect importer never breaks
  scoring.
- **Politeness:** dedicated HTTP client with a hard throttle (≈1 req/s), the configured
  user-agent, and backoff on 429/5xx — mirrors `ApiClient`. Attribute RIK data per its
  open-data terms; record attribution alongside the existing CC-BY-SA note.
- **Caching:** `api_cache`-style raw-HTML archive under `cache/ariregister/<person_id>.html`
  and a `search/<slug>.html`. Committed. `rebuild` replays these offline.
- **Parsing robustness:** tolerate Estonian diacritics, hyphenated and Cyrillic-origin
  names, and party historical name variants (resolve by registry code first).

### Component 2 — registry-code → party map

- Extend `parties` with `registry_code TEXT` (nullable, unique where present); seed the six
  seeded parties' codes in the migration. The parser maps a membership's registry code →
  `parties.short_name`; unknown codes fall back to the party name, then to inserting the
  party un-abbreviated (same non-breaking fallback philosophy as `faction_to_party`).

### Component 3 — writer + CLI

- `write_erakond_terms(conn, member, memberships, ctx)` — build `member_erakond_terms` for a
  member from parsed memberships, chronologically, mapping each to a `party_id`. Idempotent
  under the TRUNCATE-first cutover.
- New CLI command **`erakond`** (parallel to `photos`): for each member, resolve → fetch →
  write. Added to the daily GitHub Actions cron **after** `refresh-members`. Reports matched
  / unmatched counts.
- `rebuild` extended to replay the committed `cache/ariregister/` and populate
  `member_erakond_terms` offline.

### Component 4 — schema migration `0003_erakond.sql`

- `ALTER TABLE member_party_terms RENAME TO member_faction_terms;` (+ rename its indexes;
  semantics unchanged — the fraktsioon a member sits in, `party_id = NULL` = non-attached).
- `CREATE TABLE member_erakond_terms (id, member_id FK, party_id FK NULL, started_on,
  ended_on, source TEXT DEFAULT 'ariregister', CHECK ended_on >= started_on)` with a
  `(member_id, started_on)` index.
- `ALTER TABLE parties ADD COLUMN registry_code TEXT;` + seed codes for RE/EKRE/KE/E200/
  SDE/I.
- Recreate the discipline views (below). Record `0003` in `schema_migrations`.

### Component 5 — discipline views rework

All in `0003_erakond.sql`, replacing the `member_vote_alignment` / `member_discipline`
definitions from `0001`. The definitions stay in SQL (CLAUDE.md "Core metric" updated to
match).

- **Faction party line** — per `(vote, party)`, strict majority (yes/no/abstain) among
  **faction members** of that party at the vote date (from `member_faction_terms`,
  `party_id NOT NULL`, date-contained), excluding procedural votes.
- **Scoring party** — per `(member, vote)`:
  `COALESCE(faction party at vote_date, erakond party at vote_date)`. Faction party from
  `member_faction_terms` (non-NULL); erakond party from `member_erakond_terms`, date-bounded
  by the membership's join/leave dates (open-ended where `ended_on IS NULL`).
- **`member_vote_alignment`** — for each member-ballot: look up the scoring party; look up
  that party's line; `aligned` if `member_choice = line`, `defection` if it differs, `NULL`
  (excluded) if the line has no strict majority, the member's choice is `absent`, the vote
  is procedural, or the member has no scoring party. **Self-exclusion:** subtract the
  member's own ballot from the line totals **only** when the member is a faction member of
  the scoring party (erakond-scored members are not in the faction totals, so nothing is
  subtracted — handled by the join, not a special case).
- **`member_discipline`** — unchanged shape (`counted_votes`, `aligned_votes`,
  `defections`). `discipline_score = aligned/(aligned+defection)`.
- **`member_current_party`** (web) — return the member's **current scoring party**
  (faction party if currently in one, else current erakond), plus `in_faction BOOLEAN` and,
  where it differs, the erakond. Drives the corrected table badge and feeds C.

### Component 6 — web (minimal)

- `getMemberDiscipline` query reads `member_current_party`'s scoring party so the badge
  shows e.g. RE for Laneman; the 17 now carry real scores and sort correctly.
- Expose `in_faction` (and erakond-vs-faction party) in the row type for C to consume. No
  new components or visuals in this slice. Optionally a subtle textual marker may be added
  in C, not here.

## Data flow

1. `votings` ingest (unchanged) → `member_faction_terms` (chronological, from
   `voters[].faction`).
2. `erakond` command → resolve each member in äriregister → `member_erakond_terms`.
3. Views derive scoring party (faction-first, erakond-fallback) and the faction-defined line
   → `member_discipline`.
4. Web reads `member_discipline` + `member_current_party`.

## Error handling

- **Unmatched member in RIK** → no erakond terms; if also non-attached → excluded (correct).
  Logged in the `erakond` command summary; never aborts the run.
- **Ambiguous search (same name + DOB)** → extremely unlikely; if it occurs, prefer the
  candidate whose history includes a seeded party; otherwise log and skip (no erakond).
- **Unknown registry code** → insert party un-abbreviated (non-breaking), log for follow-up.
- **RIK HTML structure change** → parser raises a clear error; tests on committed fixtures
  catch this in CI before a cron run writes garbage.
- **Network unavailable (sandbox)** → `erakond` live run is skipped; `rebuild` from the
  committed cache always works offline.

## Testing

- **Fixtures** under `apps/scraper/fixtures/ariregister/`: search + history HTML for each of
  the four cases — (a) faction member who is also a party member, (b) faction member who is
  not a party member, (c) non-attached but a party member (Laneman), (d) non-attached with
  no party — plus a **name-collision** search fixture (multiple same-name people, resolved
  by DOB).
- **Unit tests:** search/history HTML parsing; name+DOB matching incl. collision and
  no-match; registry-code → party mapping incl. unknown code; chronological erakond-term
  building.
- **SQL view tests:** a small fixture DB asserting each of the four cases scores as the
  truth table says, that the faction-only line ignores non-attached party members, and that
  self-exclusion fires only for faction members.
- Existing offline scraper tests and `next lint`/typecheck/vitest stay green.

## Cutover

Follow the established protocol (memory: Neon prod cutover):

1. Apply `0003` on an **isolated Neon branch**; `rebuild` (faction terms) + `erakond`
   (erakond terms) from committed caches; verify the 17 now score and spot-check Laneman =
   RE with the expected alignment.
2. Record the **new** discipline totals (they change — ~17 members re-enter scoring; this is
   intended) and a before/after diff for the audit log.
3. Apply in place: `0003`, TRUNCATE the writer-managed tables, `rebuild`, `erakond`,
   `photos`. Commit the validated `cache/ariregister/` archive.
4. Web surfaces corrected scores via ISR within ~1h.

## Out of scope (this slice)

- The visible "party member, not in faction" indicator and any per-member faction-vs-party
  timeline visuals — **slice C** (this slice only produces the data + `in_faction` field).
- A user toggle to exclude non-faction members from scoring — way-later per the roadmap.
- Backfilling erakond history across past Riigikogu terms — v1.0 historical backfill.
- Switching the procedural discriminator to `type.code` — unrelated deferred cleanup.
