# v0.2 — active members + cycle label (design)

**Date:** 2026-06-15
**Status:** approved approach, spec under review
**Scope:** A small completeness follow-up after C. Distinguish members who are **currently
sitting** from those who **voted during this term but are no longer active** (e.g. Riina
Solman, now `active: false`), enrich the latter from the per-member API endpoint, surface
them greyed-but-visible in the list with a note on their page, and add a light **"XV
Riigikogu"** cycle label. `apps/scraper` (a small ingestion reconciliation + migration
`0004`) and `apps/web` (presentational).

Builds on A2 (member enrichment) and C (member-detail page). Discipline numbers do not change.

---

## Why

The Riigikogu `/api/plenary-members` **list** endpoint returns only currently-active members
(101). A member who voted this term but has since left the active bench — taken a ministerial
post, resigned, etc. — is dropped from the list and their committee/district associations are
cleared. We still have them in the DB because they appear in the **votings** data (every voter
gets a bare member row via `write_voting`), but they never get enriched (their bio comes only
from the list), and the UI doesn't distinguish them. The only current case is **Riina Solman**
(DB has 102 members, the list has 101): her page shows no bio because we never fetched it,
even though the API's individual endpoint has it.

This is the "active/inactive within the current cycle" piece of the broader election-cycle
roadmap; multi-cycle grouping, historical backfill, and all-time stats remain v1.0/post-1.0.

## Locked decisions (from brainstorming, 2026-06-15)

1. **List treatment: greyed in-place, still ranked.** Inactive members stay in the single
   sortable ranking (their cycle discipline still counts) but are visually muted with a
   subtle `endine` ("former") tag.
2. **Cycle label now:** a static `XV Riigikogu · 2023–` subheading (one term, no selector).
   Multi-cycle UI + backfill stay at v1.0.
3. **`active` is a simple per-member boolean** (current term). No membership-period history
   this slice (YAGNI; that's the v1.0 multi-term model).
4. Inactive marker is a subtle muted tag, not a loud badge; the detail-page note is just
   "no longer active" (no end date this slice — deferred, additive).

## Data model — migration `0004_active.sql`

```sql
ALTER TABLE members ADD COLUMN IF NOT EXISTS active BOOLEAN NOT NULL DEFAULT true;
```

`active` reflects current sitting status: members in the `/api/plenary-members` list are
`true`; members present in the DB but absent from the list are `false`.

## Ingestion — reconciliation + per-member enrichment fallback (scraper)

The `members` command (`_refresh_members`) and the offline `rebuild` gain a reconciliation
step:

1. **Listed members** (as today): `write_member` with `active = COALESCE(m.active, true)`.
2. **Gap members** — uuids present in the DB (from votings) but NOT in the list: for each,
   fetch the **individual** record `GET /api/plenary-members/{uuid}`, run the same
   `write_member` enrichment, and set `active = false`. (Committees/districts will usually be
   empty — the API has cleared them for inactive members — which `write_member` already
   tolerates.)
3. The individual responses are archived to the committed cache
   `apps/scraper/cache/api/plenary-members-extra.json` (a JSON array of raw member records),
   so offline `rebuild` reproduces them with no network.

`write_member` gains an `active: bool` parameter (default `true`); `db.enrich_member` (or a
small dedicated setter) writes the `active` column. The live command discovers gap uuids by
querying the DB for members whose `riigikogu_id` is not in the freshly-fetched list; `rebuild`
processes the list (`active=true`) then the extras cache (`active=false`).

`ApiVoteCache` gains `write_members_extra(list)` / `read_members_extra()` mirroring the
existing members snapshot methods.

## Web — list, detail, cycle label

- **Queries:** `getMemberDiscipline` and `getMemberDetail` add `active: boolean` to their row
  types.
- **Members table:** inactive rows remain in the sorted ranking but render **muted** (reduced
  opacity on avatar/name/discipline bar) with a small `endine` tag after the party badge.
  Sorting/filtering are unchanged.
- **Detail page (`MemberHeader` / a small sibling):** when `!active`, show a muted note —
  ET *"Oli XV Riigikogu liige · ei ole enam aktiivne"*, EN *"Was a member of the XV Riigikogu
  · no longer active"*. The header avatar/name are gently de-emphasized. **No membership end
  date this slice** — we don't persist one yet; appending the date (from
  `plenaryMembership.endDate`) is a deferred nicety that's purely additive.
- **Cycle label:** a `XV Riigikogu · 2023–` subheading rendered under the members-list heading
  and on the detail page, from a static i18n string (one term today).

## i18n

New `et` (authoritative) + `en` keys: `members.cycle` (the "XV Riigikogu · 2023–" label),
`members.former` / `table.former` (the `endine` tag), and `memberDetail.formerMember` (the
detail-page note).

## Testing

- **Scraper unit tests:** the reconciliation logic — a fixture member present in the list →
  `active=true`; a fixture gap member (Solman-shaped: in DB/votings, absent from list, bio in
  the individual record) → enriched + `active=false`; `ApiVoteCache` extra round-trip.
- **Web:** `active` is threaded into the existing row types (typecheck); the muted/tag and the
  detail note are thin presentational changes verified by the production build + a live
  eyeball (as with B/C). No new pure-logic module is needed.

## Cutover

Apply `0004` to prod; re-run `members` live (fetches Solman's individual record, sets `active`
across all members); commit the `plenary-members-extra.json` cache; redeploy `apps/web`.
Additive — discipline totals are unchanged (only Solman's bio populates and the active flags
are set).

## Out of scope (this slice)

- Multi-cycle discipline grouping, a cycle selector, and **historical backfill across past
  terms** — v1.0 (already roadmapped). The schema already term-links votes
  (`vote → sitting → term`), so this stays additive.
- All-time cross-cycle stats — post-1.0.
- Modeling *why* a member is inactive (minister vs resigned vs deceased) beyond the
  `active` flag and any end date — not needed now.
