# v0.4-A — Faction rollups (design)

**Date:** 2026-06-16
**Status:** approved (brainstorming) — ready for writing-plans
**Version:** v0.4 (party/faction rollups), sub-slice A of two.

## Context

v0.4's roadmap theme is "Party / faction rollups: party cohesion view; faction-level
discipline, attendance, and topic emphasis; committee-level views." That spans two very
different kinds of work — new metrics on data we already have, versus a brand-new ingestion
domain (committee votes) — so v0.4 is split into two sub-slices, each with its own
spec → plan → implement cycle:

- **v0.4-A (this doc)** — faction rollups on **existing plenary data**. No new ingestion.
- **v0.4-B (later)** — ingest committee votes (`/api/votings/committees`): new schema,
  CLI command, committed cache, then committee-level discipline views + UI.

This spec covers **v0.4-A only**.

### Decisions taken during brainstorming

1. **Cohesion = aggregate discipline.** Reuse the shipped metric verbatim, rolled up to the
   faction; do **not** introduce the Agreement Index (Rice) or any new scoring logic.
   Consequence: "cohesion" and "faction-level discipline" are the **same number** — one
   loyalty metric per faction, not two.
2. **Attendance basis = all votes, including procedural** (presence checks are themselves an
   attendance signal). This differs from the discipline denominator, which excludes
   procedural types.
3. **No topic panel in v0.4-A.** Per-faction "topic emphasis" from voting data is not
   meaningful (every faction votes on the same plenary votes); the genuine voting-topic
   dimension (per-topic cohesion) is deferred, and real "emphasis" belongs to v0.5
   (speeches).
4. **Pages: comparison index + per-faction detail**, detail kept static (roster + summary,
   no bespoke chart).
5. **Index layout: comparison card grid** (one card per faction).
6. **Detail depth: roster + summary** (header metrics, member roster ranked by discipline
   with most/least-loyal highlighted, defections summary; no cohesion-over-time chart).

## Scope

Two new routes, both read-only over existing data — **no new ingestion, no change to the
discipline definition**:

- `/fraktsioonid` — comparison index, card grid of the six factions.
- `/fraktsioonid/[slug]` — per-faction detail.

**Out of scope (explicitly deferred):** committee votes (v0.4-B), topic emphasis /
per-topic cohesion, cohesion-over-time trend chart, speeches, any change to scoring.

## Metrics

All factions in scope are the six seeded parliamentary factions (RE, EKRE, KE, E200, SDE, I).

### Cohesion (= faction discipline) — one number

Aggregate the existing per-ballot alignment (`ballot_alignment` materialized view), grouped
by `party_id` (party-at-time, already baked into the matview):

```
cohesion = Σ aligned / Σ counted
```

where `counted` / `aligned` use the **exact same predicate as `member_discipline`**:
non-procedural ballots that have a party majority line (`party_majority_choice IS NOT NULL`)
and a registered member choice (`member_choice IN ('yes','no','abstain')`); `aligned` adds
`member_choice = party_majority_choice`. This is `member_discipline`'s expression grouped by
party instead of by member. Also surface total **defections** (`counted − aligned`).

Because it reads `ballot_alignment` (already indexed and refreshed after every ingest), this
is fast and needs no new refresh step.

### Attendance

Per ballot: **present = `choice != 'absent'`**. Denominator = **all** votes, **including
procedural** (presence checks, agenda votes). Each ballot is attributed to the member's
**faction-at-time** via a plain join `ballots ⋈ votes ⋈ member_faction_terms` on the date
range (`mft.started_on <= v.voted_at::date < COALESCE(mft.ended_on, ∞)`, `party_id NOT NULL`):

```
attendance = count(choice != 'absent') / count(*)   -- grouped by faction-at-time party_id
```

This cannot read `ballot_alignment` (which excludes procedural and absent ballots), so it is
its own aggregate over `ballots`. It is a single join over ~50k rows (no correlated
subqueries) — cheap.

**Note on bloc difference:** attendance is attributed via `member_faction_terms` (faction
bench only), whereas cohesion's bloc (`ballot_alignment` by `party_id`) also includes
erakond-only members scored against the party as a fallback. So the two metrics' denominators
can differ by the handful of erakond-only members. This is intentional and acceptable —
attendance is a "did the bench show up" measure; the difference is at most ~4 members.

### Member count

Current members whose **scoring party = X** (`member_current_party.party_id = X`). The headline
count is this bloc. Erakond-only members (party member but **not** in the faction bench,
`in_faction = false`) are included in the cohesion bloc but flagged on the detail roster with
the existing "party member · not in faction" chip. The inherent term-vs-now mismatch
(cohesion is over the whole term; member count is current) is acceptable and normal.

## Data layer

New light migration **`packages/db/migrations/0007_faction_rollup.sql`** adding two read-only
**plain views** (no matview — both are cheap; nothing new to refresh beyond the existing
`ballot_alignment`):

- `faction_discipline` — `(party_id, counted_votes, aligned_votes, defections)` aggregating
  `ballot_alignment` by `party_id`, using the `member_discipline` predicate. (`cohesion` is
  computed in the query/UI as `aligned_votes / counted_votes` to keep the view integer-clean,
  matching how the existing `member_discipline` view exposes raw counts and `queries.ts`
  derives the ratio.)
- `faction_attendance` — `(party_id, present_ballots, total_ballots)` from
  `ballots ⋈ votes ⋈ member_faction_terms` (faction-at-time), counting present vs all.

Keeping metric definitions in SQL follows the project convention (discipline logic lives in
views). The web app stays read-only.

Migration is applied with the existing `migrate` CLI command; it creates only views, so no
data rebuild is required.

### Web modules

- `apps/web/lib/factions-queries.ts` — read queries:
  - `getFactionComparison()` → one row per faction with cohesion, attendance, member count,
    defections, counted votes (joins `faction_discipline`, `faction_attendance`, a current
    member-count subquery over `member_current_party`, and `parties` for name/short_name).
  - `getFactionDetail(slug)` → faction header metrics + the faction's member roster (reusing
    the `member_discipline` + `member_current_party` shape already returned by
    `getMemberDiscipline`, filtered to the faction) + defections summary; `null` if the slug
    is not one of the six.
- `apps/web/lib/factions.ts` — **pure** helpers only (so it never imports `db`): faction sort
  comparators, percent formatting, slug↔party mapping, most/least-loyal selection. Unit-tested
  with vitest (mirrors `lib/topics.ts` / `lib/members.ts`).

## UI

Reuses the v0.2 design system (party-palette tokens RE/EKRE/KE/E200/SDE/I, shadcn/ui,
light/dark, Framer Motion with `prefers-reduced-motion`, Estonian-first `as-needed` routing).

### `/fraktsioonid` — comparison index (card grid)

Six faction cards. Each card: party badge + faction name, a **cohesion bar** filled in the
party color, attendance, member count, defection count, linking to the faction detail page.
Sortable by cohesion / attendance / member count; **default sort = cohesion descending**
(most cohesive first). New nav link. i18n et+en. `/factions → /fraktsioonid` redirect for
parity with the existing `/topics → /teemad`.

### `/fraktsioonid/[slug]` — per-faction detail (static)

- Header: faction name + badge, cohesion, attendance, member count, total defections.
- Member roster ranked by discipline (reuse the members-table row component, filtered to the
  faction), with most-loyal / least-loyal members highlighted; erakond-only members carry the
  "party member · not in faction" chip.
- Defections summary.
- ISR (`revalidate = 3600`), `generateStaticParams` over the six factions, `dynamicParams`
  off (unknown slug → 404).
- Slug = party `short_name` lowercased (`/fraktsioonid/ekre`, `/fraktsioonid/e200`, …).

## Consistency / non-goals

- The discipline definition in `0003`/`0006` is **untouched**; v0.4-A only aggregates and
  displays existing per-ballot alignment. Faction cohesion numbers are derivable from member
  scores already shown.
- No write path from the web app (unchanged).
- No emoji in code/commits/docs (project convention).
