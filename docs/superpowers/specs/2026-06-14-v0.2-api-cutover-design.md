# v0.2 / A1 — API ingestion cutover (design)

**Date:** 2026-06-14
**Status:** approved approach, spec under review
**Scope:** Sub-project A1 of v0.2. Rebase votes + members ingestion from HTML scraping
onto the official Open Data API (`api.riigikogu.ee`), with API-native models, a raw-JSON
archive, and a parity-gated cutover. **No new DB tables.** The structural schema
(`0002_*`: terms, committees, districts, sessions/sittings) and member enrichment are
the next spec (A2) and are explicitly out of scope here.

---

## Why

v0.1 ingests by parsing three HTML pages (vote list, vote detail, members). The official
API exposes the same data as stable-UUID JSON, is CC-BY-SA, and carries fields the HTML
never did (real vote-type codes, per-ballot faction-at-time with a faction UUID, member
bios, districts, photos). The roadmap (governing decision 1) is to migrate ingestion to
the API. This spec does the votes+members cutover — the foundation every later v0.2 slice
reads from — and proves it reproduces the current discipline scores before switching over.

Decision (2026-06-14, this session): go **API-native** (model the API faithfully, not an
adapter into the old types) and **discard the HTML ingestion path entirely**. HTML parsers
are reintroduced only if a concrete data gap in the API is found; probing votes, members,
and per-ballot faction shows no such gap.

## Verified API shapes (probed live, 2026-06-14)

- `GET /api/votings?startDate=&endDate=` → array of **sitting groups**:
  `{ uuid, title, membership, sittingDateTime, votings: [ { uuid, votingNumber,
  type{code,value}, description, startDateTime, endDateTime, present, absent, inFavor,
  against, neutral, abstained } ] }`. `type.code` is a real discriminator, e.g.
  `KOHALOLEKU_KONTROLL`, `AVALIK`. `description` ≈ the old title ("Kohaloleku kontroll",
  "Lõpphääletus").
- `GET /api/votings/{voting-uuid}` → one voting with `voters: [ { uuid, fullName,
  firstName, lastName, active, faction{uuid,name}, decision{code,value}, upn } ]` (101
  rows). `faction` is the member's faction **at the time of the vote** — the signal the
  party-term tracking needs. `decision.code` ∈ {POOLT, VASTU, ERAPOOLETU, PUUDUB,
  "did not vote"} (exact codes confirmed in implementation).
- `GET /api/plenary-members` → 101 members: `{ uuid, fullName, firstName, lastName,
  active, email, phone, electoralDistrictHistory[], photo{uuid,...},
  plenaryMembership{membershipNumber,startDate,endDate,role,jobTitle}, factions[] }`.
- `GET /api/memberships/current` → `{ number:15, startDate, endDate, sessions[] }` (used
  in A2, not here).

The non-attached faction is `"Fraktsiooni mittekuuluvad Riigikogu liikmed"`, which the
existing `normalize_faction` ("mittekuuluv" → None) already handles.

## Architecture

The cutover swaps the **source** (HTML → API) and the **in-memory model** (HTML-era types
→ API-native types). It preserves three things deliberately:

1. **DB schema** — `0001_initial.sql` tables and discipline views are unchanged. The API
   writer targets the same `members / parties / member_party_terms / votes / ballots`.
2. **Party-term transition logic** — the v0.1 fix (close the open term and open a new one
   on any party ↔ non-attached change, fed votes in chronological order) is the valuable,
   tested core. It is retained; it now consumes API-native voters instead of HTML ballots.
3. **The discipline definition** — to keep the parity gate honest, `votes.vote_type_slug`
   is populated exactly as today (derive from the voting `description` via the existing
   `title_to_slug`), so procedural exclusions and scores are identical. The cleaner
   `type.code` discriminator is carried in-memory but not yet persisted; switching the
   discriminator to it is a documented follow-up after parity passes.

### Components

- **`api_client.py`** (new) — JSON sibling of `client.py`. Base `https://api.riigikogu.ee`,
  **hard 1 req/s** throttle (reuse `client.py`'s `_respect_delay` shape), retry/backoff on
  429/5xx, CC-BY-SA User-Agent. Returns parsed JSON. (We hit a real 429 during research;
  the limiter is strict.)
- **`api_models.py`** (new) — API-native pydantic models: `SittingGroup`, `VotingSummary`,
  `Voting`, `Voter`, `Faction`, `PlenaryMember` (+ nested `Photo`, `PlenaryMembership`).
  Faithful to the JSON (carry `uuid`s and fields A2 will use, e.g. faction uuid, photo
  uuid, district history), `extra='ignore'` so unmodeled fields don't break ingest.
- **`api_parse.py`** (new) — pure functions JSON → models, and the two mappings the writer
  needs: `Voting + voters → (vote row, ballots)` and `PlenaryMember → member row`.
  Decision-code → choice map lives here (POOLT→yes, VASTU→no, ERAPOOLETU→abstain,
  PUUDUB→absent, did-not-vote→neutral).
- **`api_cache.py`** (new) — raw-JSON archive under `apps/scraper/cache/api/`:
  - `votings.jsonl` — one **raw** `/api/votings/{uuid}` response per line, resumable by
    uuid (`has(uuid)` reflects disk, mirroring today's `VoteCache`).
  - `plenary-members.json` — raw `/api/plenary-members` snapshot.
  Storing raw (not distilled) JSON means A2's enrichment reads the same archive with no
  re-fetch. Chronological replay still happens at rebuild time (sort by voting start).
- **`writer.py` / `db.py`** — the party-term + upsert logic moves to consume API-native
  models. DB functions (`upsert_member`, `set_member_party`, `upsert_vote`,
  `replace_ballots`, `upsert_party`) keep their table contract; signatures adjust to the
  new model types. Parties stay keyed on the seeded `short_name` (RE/EKRE/KE/E200/SDE/I)
  via faction-name mapping; faction `uuid` is carried but persisting it is A2.
- **`cli.py`** — commands become API-sourced:
  - `backfill --from --to [--cache-only]` — fetch sitting groups by date, then each
    voting's detail, archive raw, write DB.
  - `daily` — yesterday's votings (the GH Actions cron target).
  - `members` — refresh from `/api/plenary-members`.
  - `rebuild` — offline replay from `cache/api/`.
  The HTML parsers (`parsers/vote_*`, `parsers/members.py`), `client.py`, the distilled
  `VoteCache`, and their fixtures/tests are **removed** once parity passes (see Cutover).

### Data flow

```
/api/votings?date  ->  SittingGroup[] (voting summaries)
  for each voting summary not already archived:
    /api/votings/{uuid}  ->  raw JSON  -> api_cache.votings.jsonl
                                       -> Voting + voters (api_models)
                                       -> (vote, ballots) (api_parse)
                                       -> writer: upsert member/party/term, vote, ballots
/api/plenary-members   ->  raw JSON   -> api_cache.plenary-members.json
                                       -> PlenaryMember[]  -> writer: upsert members + current faction
DB (unchanged schema) -> member_discipline view -> apps/web (unchanged)
```

## Parity gate (the cutover criterion)

Before switching production, prove the API reproduces current scores:

1. Create a **Neon branch** off production (isolated, copy-on-write).
2. Into the branch, re-ingest the **same 1-year window** (`--from 2025-06-14`) via the API.
3. Run `member_discipline` on the branch and diff per-member
   `(counted_votes, aligned_votes, defections)` against current production.
4. **Pass** = every member matches exactly, **or** each remaining difference is traced to
   the API legitimately having more/fewer votes than the HTML scrape (e.g. votes the HTML
   pagination missed) and judged correct. Record the diff summary in `progress-log.md`.
5. Only then: re-ingest production from the API archive, flip the GH Actions `daily` cron
   to the API path, set the missing GH Actions `DATABASE_URL` secret (the still-open v0.1
   item), redeploy `apps/web` (unchanged), and delete the HTML ingestion code.

## Testing

- **Fixtures** (committed): `apps/scraper/fixtures/api/{voting_detail,plenary_members,
  votings_list}.json` — real (trimmed) API responses.
- **Parser tests**: `api_parse` maps the voting fixture to the expected `(vote, ballots)`
  (choice mapping, non-attached → None, slug == `title_to_slug(description)`); maps the
  members fixture to expected member rows.
- **Offline rebuild test**: `rebuild` from a committed `cache/api/votings.jsonl` reproduces
  votes + ballots with no network (mirrors today's offline guarantee).
- **Rate-limit assertion**: a test/log confirming the client never issues two requests
  inside 1 s.
- Existing discipline-view behavior is covered by the parity gate, not re-tested here.

## Files

- New: `api_client.py`, `api_models.py`, `api_parse.py`, `api_cache.py`,
  `writer.py` (extracted party-term/upsert logic), `fixtures/api/*`, `tests/test_api_*`.

  (`faction_to_party` / `normalize_faction` retained; HTML-only types removed),

  `cache/members.json`, `fixtures/{vote_list,vote_detail,members}.html` and their tests.

## Out of scope (→ A2 and later)

- `0002_*` structural tables: `riigikogu_terms`, `committees` + `member_committee_terms`,
  `electoral_districts` + `member_district_terms`, `sessions` + `sittings`.
- Member enrichment columns (photo_uuid, district, birth year, seniority, email/phone).
- Persisting faction uuid / vote `type.code` / vote→sitting FK; switching the procedural
  discriminator from title-slug to `type.code`.
- Backfill across past terms; Eurovoc; any UI change (B / C).

## Open questions / risks

- **Exact `decision.code` set** — confirm all codes (incl. "did not vote" vs absent) on
  first real fetch; the choice map is the only place a wrong code corrupts scores.
- **Faction-name stability** — `faction_to_party` maps by name; the API also gives a
  stable faction `uuid`. If a name diverges from the seed strings, mapping by uuid is the
  robust fix (carry the uuid now, switch the mapping key in A2 if needed).
- **Vote count parity** — if the API and HTML disagree on which votes exist in the window,
  the parity gate surfaces it; resolve before cutover rather than papering over.
