**Status:** approved approach, spec under review
**Scope:** Sub-project A1 of v0.2. Rebase votes + members ingestion from HTML scraping

## Why

v0.1 ingests by parsing three HTML pages (vote list, vote detail, members). The official
API exposes the same data as stable-UUID JSON, is CC-BY-SA, and carries fields the HTML
never did (real vote-type codes, per-ballot faction-at-time with a faction UUID, member
bios, districts, photos). The roadmap (governing decision 1) is to migrate ingestion to
the API. This spec does the votes+members cutover â€” the foundation every later v0.2 slice
reads from â€” and proves it reproduces the current discipline scores before switching over.

Decision (2026-06-14, this session): go **API-native** (model the API faithfully, not an
adapter into the old types) and **discard the HTML ingestion path entirely**. HTML parsers
are reintroduced only if a concrete data gap in the API is found; probing votes, members,
and per-ballot faction shows no such gap.

## Architecture

The cutover swaps the **source** (HTML â†’ API) and the **in-memory model** (HTML-era types
â†’ API-native types). It preserves three things deliberately:

1. **DB schema** â€” `0001_initial.sql` tables and discipline views are unchanged. The API
   writer targets the same `members / parties / member_party_terms / votes / ballots`.
2. **Party-term transition logic** â€” the v0.1 fix (close the open term and open a new one
   on any party â†” non-attached change, fed votes in chronological order) is the valuable,
   tested core. It is retained; it now consumes API-native voters instead of HTML ballots.
3. **The discipline definition** â€” so the post-port parity check stays meaningful,
   `votes.vote_type_slug` is populated exactly as today (derive from the voting
   `description` via the existing `title_to_slug`), so procedural exclusions and scores
   are computed identically. The cleaner `type.code` discriminator is carried in-memory
   but not yet persisted; switching the discriminator to it is a documented follow-up.

### Components

- **`api_client.py`** (new) â€” JSON sibling of `client.py`. Base `https://api.riigikogu.ee`,
  **hard 1 req/s** throttle (reuse `client.py`'s `_respect_delay` shape), retry/backoff on
  429/5xx, CC-BY-SA User-Agent. Returns parsed JSON. (We hit a real 429 during research;
  the limiter is strict.)
- **`api_models.py`** (new) â€” API-native pydantic models: `SittingGroup`, `VotingSummary`,
  `Voting`, `Voter`, `Faction`, `PlenaryMember` (+ nested `Photo`, `PlenaryMembership`).
  Faithful to the JSON (carry `uuid`s and fields A2 will use, e.g. faction uuid, photo
  uuid, district history), `extra='ignore'` so unmodeled fields don't break ingest.
- **`api_parse.py`** (new) â€” pure functions JSON â†’ models, and the two mappings the writer
  needs: `Voting + voters â†’ (vote row, ballots)` and `PlenaryMember â†’ member row`.
  Decision-code â†’ choice map lives here (POOLTâ†’yes, VASTUâ†’no, ERAPOOLETUâ†’abstain,
  PUUDUBâ†’absent, did-not-voteâ†’neutral).
- **`api_cache.py`** (new) â€” raw-JSON archive under `apps/scraper/cache/api/`:
  - `votings.jsonl` â€” one **raw** `/api/votings/{uuid}` response per line, resumable by
    uuid (`has(uuid)` reflects disk, mirroring today's `VoteCache`).
  - `plenary-members.json` â€” raw `/api/plenary-members` snapshot.
  Storing raw (not distilled) JSON means A2's enrichment reads the same archive with no
  re-fetch. Chronological replay still happens at rebuild time (sort by voting start).
- **`writer.py` / `db.py`** â€” the party-term + upsert logic moves to consume API-native
  models. DB functions (`upsert_member`, `set_member_party`, `upsert_vote`,
  `replace_ballots`, `upsert_party`) keep their table contract; signatures adjust to the
  new model types. Parties stay keyed on the seeded `short_name` (RE/EKRE/KE/E200/SDE/I)
  via faction-name mapping; faction `uuid` is carried but persisting it is A2.
- **`cli.py`** â€” commands become API-sourced:
  - `backfill --from --to [--cache-only]` â€” fetch sitting groups by date, then each
    voting's detail, archive raw, write DB.
  - `daily` â€” yesterday's votings (the GH Actions cron target).
  - `members` â€” refresh from `/api/plenary-members`.
  - `rebuild` â€” offline replay from `cache/api/`.
  The HTML parsers (`parsers/vote_*`, `parsers/members.py`), `client.py`, the distilled
  `VoteCache`, and their fixtures/tests are **removed** once parity passes (see Cutover).

### Data flow

```
/api/votings?date  ->  SittingGroup[] (voting summaries)
  for each voting summary not already archived:
    /api/votings/{uuid}  ->  raw JSON  -> api_cache.votings.jsonl
                                       -> Voting + voters (api_models)
                                       -> (vote, ballots) (api_parse)
                                       -> writer: upsert member/party/term, vote, ballots
/api/plenary-members   ->  raw JSON   -> api_cache.plenary-members.json
                                       -> PlenaryMember[]  -> writer: upsert members + current faction
DB (unchanged schema) -> member_discipline view -> apps/web (unchanged)
```

## Cutover, then parity check (validation, not a gate)

Decision (2026-06-14): do the port first, validate parity after â€” the parity check is a
post-port sanity pass, not a precondition for cutting over or deleting the HTML code.

**Port + cutover:**
1. Build the API client / models / parser / cache / writer and the API-sourced CLI.
2. Re-ingest the 1-year window (`--from 2025-06-14`) via the API into production, flip the
   GH Actions `daily` cron to the API path, set the missing GH Actions `DATABASE_URL`
   secret (the still-open v0.1 item), redeploy `apps/web` (unchanged).
3. Delete the HTML ingestion code (parsers, `client.py`, distilled cache, fixtures/tests).

**Post-port parity check (validation):**
4. Re-run the old HTML-derived `member_discipline` numbers (captured before deletion, or
   from the prior production snapshot) against the new API-derived numbers; diff per-member
   `(counted_votes, aligned_votes, defections)`.
5. Any non-trivial difference is investigated and **fixed forward** â€” traced to a real
   data difference (API has more/fewer votes than the HTML scrape) and judged correct, or
   fixed if it's a porting bug. Record the diff summary in `progress-log.md`.

Capture the current per-member discipline numbers to a file **before** deleting the HTML
path, so step 4 has a baseline to diff against.

## Testing

- **Fixtures** (committed): `apps/scraper/fixtures/api/{voting_detail,plenary_members,
  votings_list}.json` â€” real (trimmed) API responses.
- **Parser tests**: `api_parse` maps the voting fixture to the expected `(vote, ballots)`
  (choice mapping, non-attached â†’ None, slug == `title_to_slug(description)`); maps the
  members fixture to expected member rows.
- **Offline rebuild test**: `rebuild` from a committed `cache/api/votings.jsonl` reproduces
  votes + ballots with no network (mirrors today's offline guarantee).
- **Rate-limit assertion**: a test/log confirming the client never issues two requests
  inside 1 s.
- Existing discipline-view behavior is covered by the parity gate, not re-tested here.

## Open questions / risks

- **Exact `decision.code` set** â€” confirm all codes (incl. "did not vote" vs absent) on
  first real fetch; the choice map is the only place a wrong code corrupts scores.
- **Faction-name stability** â€” `faction_to_party` maps by name; the API also gives a
  stable faction `uuid`. If a name diverges from the seed strings, mapping by uuid is the
  robust fix (carry the uuid now, switch the mapping key in A2 if needed).
- **Vote count parity** â€” if the API and HTML disagree on which votes exist in the window,
  the post-port parity check surfaces it; fix forward rather than papering over.
