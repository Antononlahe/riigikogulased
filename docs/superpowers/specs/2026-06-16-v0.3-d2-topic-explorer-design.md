# v0.3 / D2 — Topic explorer UI (design)

**Date:** 2026-06-16
**Status:** approved approach, spec under review
**Scope:** The second half of v0.3 (Topics). A **dedicated, self-contained topic-explorer**
section of the web app that surfaces topic-filtered discipline on top of D1's data
(`vote_topics` view + Eurovoc tables). Plus a small **site-wide locale-routing change**
(Estonian-first URLs) requested alongside. **Read-only UI** — no migration, no scraper change,
no change to any discipline scoring SQL. Discipline numbers are unchanged everywhere.

Builds on D1 (live in prod): the `vote_topics` view resolves `vote_id → descriptor_edid,
microthesaurus_etid, field_efid`; `eurovoc_fields`/`eurovoc_microthesauri`/`eurovoc_descriptors`
carry bilingual labels.

---

## Why

Governing decision 3 of the roadmap: topic categorization uses official Eurovoc tags. D1 landed
the data + the `vote_topics` foundation; D2 is the user-facing payoff — "how loyal is each MP on
a given topic, and what did parliament actually vote on under that topic." Because Eurovoc sits
conceptually far from the core discipline metric, D2 is built so it can be **removed later as a
small, mechanical diff** if the topic angle is dropped (see Isolation).

## Governing constraint — Isolation / removability

Everything topic-related lives behind a clean seam. Removing D2 = delete these and nothing else:

- `apps/web/app/[locale]/teemad/**` (route tree)
- `apps/web/components/topics/**` (components)
- `apps/web/lib/topics.ts` (data access) + `apps/web/lib/topics.test.ts`
- the `topics` namespace in `apps/web/messages/{et,en}.json`
- one nav link in `apps/web/components/site-header.tsx`
- the `/topics → /teemad` redirect entry in `apps/web/next.config.ts`

Hard rules: **no migration; no edit to `member_discipline`, `member_vote_alignment`,
`member_current_party`, or any scoring SQL; no topic UI on the homepage table or the member
detail page.** D2 only *reads* `vote_topics` and aggregates with the same expression
`member_discipline` already uses. The locale-routing change (below) is the one deliberately
non-isolated piece, called out explicitly.

## Data reality (verified on prod, 2026-06-16)

Scoped to **counted** votes (`party_majority_choice IS NOT NULL`, choice in yes/no/abstain, not
procedural) joined to `vote_topics`:

- **394** distinct scored votes carry a topic (of 598 total votes; the rest are procedural or
  bill-less — correctly topicless).
- **Descriptor** level: **557** descriptors have scored votes — **100%** coverage of the 394
  votes — but the distribution is long-tailed: **258 have exactly 1 vote**, **201 have ≥3**,
  **101 have ≥5**, **30 have ≥10**.
- **Field** level: only **17** of 21 fields appear and they cover just **141/394 votes (36%)** —
  the known D1 rollup limitation (narrower descriptors have no API ancestry). Top field
  (RAHANDUS/finance) = 35 votes; most fields < 20.
- **Microthesaurus** level: 39 appear, same ~36% coverage as fields (shares the rollup gap).

Two consequences drive the design: (1) **descriptor is the only complete level**, so it is the
spine; field is context only. (2) defections are rare (~0.5% overall) and per-topic counts are
small, so per-MP per-topic discipline reads ~100% for most members — **counts and actual
defections must be shown prominently** so flat scores are visibly low-N, and the index must be
**vote-thresholded** so it isn't 258 single-vote rows.

## Locked decisions (from brainstorming, 2026-06-16)

1. **Dedicated explorer**, not a homepage filter or member-page section — keeps it isolated/
   removable.
2. **Descriptor is the browsable unit.** Field/microthesaurus are shown as context labels, never
   the primary navigation axis (avoids hiding 64% of votes in an "uncategorised" bucket).
3. **Thresholds (tunable, these are the starting values):** the topic index lists descriptors
   with **≥5 scored votes** (~101 topics); within a topic, only members with **≥3 counted votes
   on that topic** are ranked.
4. **Estonian-first URLs (site-wide):** `localePrefix: "as-needed"` + `localeDetection: false`.
   Default locale **et** is unprefixed (`/`, `/members/[slug]`, `/teemad`); English is opt-in at
   `/en/...`. Bare `/` is always Estonian regardless of browser language.
5. **Topics route is Estonian-named and self-contained:** folder `teemad` → `/teemad` (et) and
   `/en/teemad` (en); a `/topics → /teemad` redirect in `next.config.ts`. No next-intl
   `pathnames` machinery (that would entangle members/home routing and reduce removability).
6. **Recursive `narrowTerms` crawl to lift field coverage is out of scope** — its own future
   ingestion slice if field faceting is ever wanted.

## Locale-routing change (B0 — site-wide, not isolated)

`apps/web/i18n/routing.ts`:

```ts
export const routing = defineRouting({
  locales: ["et", "en"],
  defaultLocale: "et",
  localePrefix: "as-needed",
  localeDetection: false,
});
```

- Result: et URLs lose the `/et` prefix; `/` and all default pages are Estonian. English at
  `/en/...` only, reached by explicit URL or the locale toggle.
- next-intl auto-redirects superfluous `/et/...` → unprefixed, and `<Link>`/locale-toggle keep
  working (they use the navigation APIs). The existing `middleware.ts` matcher already matches
  unprefixed paths — unchanged.
- `setRequestLocale` / `generateStaticParams` over `[locale]` are unaffected (both locales still
  generated; the et prefix is just elided in the URL).
- **Visible side effect beyond D2:** current `/et/...` deep links become `/...` (auto-redirected).

## Routes

- `/[locale]/teemad` — topic index. ISR `revalidate = 3600`.
- `/[locale]/teemad/[edid]` — one descriptor's detail, keyed by `descriptor_edid` (stable int
  PK). ISR `revalidate = 3600`, `dynamicParams = true`; `generateStaticParams` over the
  thresholded index (~101 × 2 locales) so the build doesn't prerender 557×2 pages — long-tail
  descriptors render on first request and cache.
- `/topics` → `/teemad` redirect (`next.config.ts` `redirects()`), locale-agnostic convenience.

## Pages

### Topic index — `/teemad`

- Header: one-line explainer + an **honesty note**: topics come from official Eurovoc tags on
  bills and cover the 394 votes tied to a bill, not all votes.
- A search box (typeahead, client-side filter over the full descriptor set so subjects below the
  ≥5 threshold are still reachable by name).
- A list of descriptors with **≥5 scored votes**, sorted by vote volume desc. Each row:
  topic name (locale-aware `text_et`/`text_en`), its field label as a muted chip **when present**
  (else nothing — no fake bucket), vote count, and total defections on the topic (its
  "contentiousness" signal). Rows link to the detail page.

### Topic detail — `/teemad/[edid]`

- **Header:** topic name; a field → microthesaurus breadcrumb when the rollup exists, else an
  explicit "ei kuulu laia valdkonda / no broad field" note; totals — N votes, N bills, N total
  defections across all members.
- **Ranked discipline table** scoped to the topic: columns `Member · Party · Votes on topic ·
  Defections · Discipline %`, sorted worst-discipline-first then most-defections (matches the
  homepage default). Only members with **≥3 counted votes on the topic** are ranked; a muted note
  states "N members had too few votes on this topic to rank." Counts sit next to every %, so a
  100%-on-3-votes row is visibly low-N. Reuses `PartyBadge` + `DisciplineBar`. Member names link
  to their detail pages.
- **Bills carrying the topic:** list of `draft_title` + date + per-bill counted/defection
  summary. Plain list — no vote-detail route exists, so no dead links.

## Data access — `apps/web/lib/topics.ts` (no migration)

Three read functions, all parameterized off the existing `vote_topics` view ⋈
`member_vote_alignment`, aggregated with the **same** counted/aligned/defection expression
`member_discipline` uses (no new scoring logic):

1. `getTopicIndex(): TopicIndexRow[]` — descriptors with ≥5 scored votes: `{ edid, name_et,
   name_en, fieldEt, fieldEn, votes, defections }`. Drives the index page **and**
   `generateStaticParams`.
2. `getTopicDetail(edid): TopicDetail | null` — header totals (votes, bills, defections), the
   field/microthesaurus breadcrumb (nullable), and the per-member rows `{ memberId, fullName,
   slug, partyShortName, inFaction, votesOnTopic, defections, disciplineScore }` filtered to
   `descriptor_edid = edid` and `votesOnTopic ≥ 3`, plus a `belowThresholdCount`.
3. `getTopicBills(edid): TopicBillRow[]` — bills carrying the descriptor (via
   `volume_topics`/`votes.draft_uuid`) with `{ draftTitle, firstDate, lastDate, counted,
   defections }`.

All pages catch DB errors and degrade (empty state), mirroring the homepage/member pages. The
locale picks `text_et` vs `text_en` (et authoritative; en falls back to et when null).

### Pure helpers — same file or `lib/topics.ts` companion, unit-tested

Threshold/format/breadcrumb logic factored into pure functions (e.g. `formatTopicLabel`,
`hasBroadField`, sort/ranking gate) with vitest, mirroring `lib/member-detail.ts`. SQL functions
themselves aren't unit-tested (no pg test harness in the repo — consistent with prior slices);
they're verified by the production build + manual spot-checks.

## i18n

New `topics` namespace in `et.json` + `en.json` (heading, explainer, honesty note, search
placeholder, column labels, "too few votes" note, breadcrumb fallback, bills heading, empty
state). ET authoritative; full en parity (build fails on `MISSING_MESSAGE`). One nav-link label
added to the existing header namespace.

## Verification

- `tsc` clean, `next lint` clean.
- vitest green for the pure helpers (extend the existing web test suite).
- Production build: prerenders the ~101 thresholded topic pages × 2 locales + both index pages,
  0 `MISSING_MESSAGE`; bundle size noted.
- Manual spot-check against a live DB: a dense topic (finance ≈ 35 votes) and a sparse one;
  confirm the ≥3 ranking gate and the low-N count display; confirm the field breadcrumb shows the
  "no broad field" note for a descriptor with `microthesaurus_etid NULL`.
- Confirm the locale change: `/` Estonian, `/en` English, `/et/...` redirects to unprefixed,
  locale toggle round-trips; **no discipline number changes** anywhere.

## Out of scope

- Recursive `narrowTerms` crawl / lifting field coverage above 36% (future ingestion slice).
- Vote-detail pages / per-vote drill-down (no such route yet; bills are a plain list).
- Topic UI on the homepage or member detail pages (kept isolated by decision 1).
- Party/faction topic cohesion rollups (v0.4 territory; the per-member table is enough for v0.3).
- Renaming `/members` to an Estonian path (not requested; only `/teemad` is Estonian-named).
